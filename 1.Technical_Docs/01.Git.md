# Git Notes

## 1. Git Pull vs Git Fetch

1. **Git Fetch** gets the latest changes from the remote repository but does not merge them into your current branch.
2. **Git Pull** gets the latest changes and merges them into your current branch.
3. Git Pull = Git Fetch + Git Merge.

---

## 2. Git Rebase vs Git Merge

1. **Git Rebase** is used to maintain a linear commit history by replaying feature branch commits on top of the target branch.
2. **Git Merge** is used to preserve the complete development history in chronological order and creates a merge commit.
3. Rebase results in a cleaner history, while Merge shows the actual branch structure and integration points.

---

## 3. How do you solve merge conflicts?

1. Identify the conflicting files.
2. Collaborate with the developers or code owners involved in the conflicting changes.
3. Review the conflicting code and understand the expected behavior.
4. Decide which implementation is correct.
5. Resolve the conflicts and test the application.
6. Commit the resolved changes and proceed with the merge.

---

## 4. What is a Git Tag? Why is it needed?

1. Git tags are used to create references or labels on important commits.
2. Common use cases:

   * Releases (v1.0, v2.0)
   * Hotfixes
   * Milestones
   * Production deployment versions
3. Tags help teams identify, track, and roll back to specific versions when needed.

---
## 5 What is git cherry-pick?
git cherry-pick is a command that allows you to select specific commits from one branch and apply them to your current branch

---
## 6 How to remove commits added to master branch ?
1. Create a new feature branch from the current master.
2. Run git reset --hard <last-good-commit> on master.
3. Force push master if needed.

---
## 7. Explain your branching strategy

1. We use protected branches for different environments: develop for Development, release for Staging/UAT, and master/main for Production.
2. Developers create feature branches for new changes.
3. Development is performed in the feature branch, and changes are merged into the develop branch through a Pull Request (PR).
4. CI/CD validations run automatically, and testing is performed in the development environment using the develop branch.
5. Once development testing is successful, changes are promoted to the release branch.
6. Extensive functional, integration, and UAT testing is performed in the release environment.
7. After successful validation and approvals, changes are merged into the master/main branch and deployed to production.
8. Release tags are created for production deployments.
9. For critical production issues, hotfix branches are created from the master/main branch.
10. After the fix is validated and deployed to production, the changes are merged back into the develop branch to keep all branches synchronized.
