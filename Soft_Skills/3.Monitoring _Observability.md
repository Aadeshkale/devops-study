# Monitoring vs Observability

## What is Monitoring?

Monitoring helps identify **when** a problem or failure has occurred in a system.

### Purpose
- Alerting and notifications
- Health checks
- Threshold-based monitoring
- Availability tracking

### Example
- CPU usage > 80%
- Pod restart count increased
- Disk space running low
- Application returning 5xx errors

### Question Answered
**"Something is wrong."**

---

## What is Observability?

Observability helps determine **why** a problem occurred by analyzing logs, metrics, and traces.

### Purpose
- Root Cause Analysis (RCA)
- Troubleshooting complex systems
- Understanding application behavior
- Dependency tracking across services

### Example
- Which service caused latency?
- Why is a request failing?
- Which database query is slow?
- Which microservice is generating errors?

### Question Answered
**"Why is it wrong?"**

---

## Monitoring vs Observability

| Feature | Monitoring | Observability |
|----------|------------|------------|
| Purpose | Detect issues | Diagnose issues |
| Focus | Known problems | Unknown problems |
| Alerting | Yes | Yes |
| Root Cause Analysis | Limited | Strong |
| Data Used | Mainly metrics | Metrics, Logs, Traces |
| Answers | What happened? | Why did it happen? |

LGTM is a popular open-source observability stack from Grafana.

## L - Loki

### Purpose
Log aggregation and log management system.

### Features
- Centralized logging
- Kubernetes log collection
- Cost-efficient storage
- Grafana integration

---

### Purpose
Visualization and dashboard platform.

### Features
- Dashboards
- Alerting
- Data visualization
- Multiple data source support

---

## T - Tempo

### Purpose
Distributed tracing backend.

### Features
- Request tracing
- Service dependency mapping
- Root cause analysis
- OpenTelemetry integration

---

## M - Mimir

### Purpose
Metrics storage and monitoring backend.

### Features
- Long-term metrics storage
- Prometheus-compatible
- Horizontal scalability
- Multi-tenant support

# OpenTelemetry

## What is OpenTelemetry?

OpenTelemetry (OTel) is an open-source observability framework used to collect:

- Logs
- Metrics
- Traces

### Components

#### OpenTelemetry SDK
Used inside applications for instrumentation.

#### OpenTelemetry Collector
Collects, processes, and exports telemetry data.

### Supported Backends
- Grafana LGTM
- Datadog
- New Relic
- Jaeger
- Prometheus

---

# Grafana Alloy

## What is Grafana Alloy?

Grafana Alloy is a unified telemetry collector built by Grafana Labs.

### Benefits

Instead of running separate agents for:

- Logs
- Metrics
- Traces

You can use a single agent:

**Grafana Alloy**

### Features

- OpenTelemetry-compatible
- Collects logs, metrics, and traces
- Simplified operations
- Lower resource consumption
- Centralized telemetry pipeline

# Typical Observability Flow

```text
Application
      ↓
OpenTelemetry SDK / Auto Instrumentation
      ↓
Grafana Alloy / OTel Collector
      ↓
+----------------------------------+
| Logs    → Loki                   |
| Metrics → Mimir / Prometheus     |
| Traces  → Tempo                  |
+----------------------------------+
      ↓
Grafana Dashboards & Alerting
```

We have monitoring and alerting at both the infrastructure and application levels.

### 1. Infrastructure Alerts

- CPU utilization > 80%
- Memory utilization > 80%
- Disk utilization > 80%
- Filesystem/inode usage threshold breaches
- High network latency or packet drops
- VM/node unreachable
- Kubernetes node NotReady
- Pod restart count increase
- Container OOMKilled events
- Database CPU, memory, storage, and connection utilization thresholds

### 2. Application Alerts

- Application downtime or health check failures
- HTTP 5xx error rate increase
- HTTP 4xx error spikes
- High response latency (P95/P99)
- API timeout errors
- Increased exception/error rates
- Failed background jobs or scheduled tasks
- Queue backlog growth (Kafka, Pub/Sub, SQS, RabbitMQ)
- SSL/TLS certificate expiry alerts
- Application pod crash loops (CrashLoopBackOff)
